> **SUPERSEDED (2026-06-27).** The "Stack A = Designed-Inline-HTML + Tailwind excellence layer" decision below was superseded. **The LLM no longer writes HTML at all.** Instead the `build` agent emits a JSON **spec** (brand tokens + an ordered list of sections), and a **deterministic component renderer** (`src/render.ts` + `src/components.ts`) builds the page from hand-built vetted components — deriving a WCAG-safe palette from the 2 brand colours and a CSS-only responsive nav. This achieves the same goals this record set out to hit — no "1998 look", self-contained pages, **zero new trust surface**, and still terminating in the existing `site_renders` chromium gate — **without** vendoring a ~120 MB Tailwind binary (that binary, `tools/setup.sh`, and `src/excellence.ts` were all removed). The Postgres content-model / owner-editing idea described below became the **live CMS** (`src/cms.ts`). The original reasoning is preserved verbatim below as a record of the decision at the time.

## Relay build-stack decision (final, build-once)

**The "1998 look" is a design-system problem, not a generator problem.** I confirmed this against the live YogaLX site (`sites/1c3c8117.../index.html`): it declares `--font-display:'Playfair Display'` and `'Open Sans'` as CSS vars but **never loads them** (an external font `<link>` would trip the gate), so they fall back to system serif/sans; hero images are the placeholder gradient `linear-gradient(135deg,#e9ecf3,#c9d2e3)` that `runner.ts:69` injects. Structurally fine, visually flat. Swapping SSGs does not fix that — a **token + utility-CSS contract that ships fonts inline** does.

**Decision: 2 stacks + a deterministic keyword router + an excellence layer that bolts onto stack #1, plus a Postgres content-model for editing. NO Astro, NO headless CMS, NO per-project `npm install` in v1.**

- **Stack A — Designed-Inline-HTML + Tailwind excellence layer** (DEFAULT). Today's per-page LLM HTML build, upgraded by a deterministic post-build compile-and-inline of the vendored Tailwind standalone binary. Handles the bulk of briefs: marketing/landing, brochure, portfolio, store-catalog.
- **Stack B — Eleventy** for blog/content + docs only. Agents emit Markdown (which MiniMax does far better than 8KB of hand-written HTML); the SSG owns layout/nav/RSS. Served from a pre-installed cached template with a **shared symlinked `node_modules`** — never per-project install. Built in ~0.6s, output is byte-structurally identical to today's (inlined `<style>`, 0 external https).
- **Excellence layer** = vendored `tailwindcss-linux-x64` (I downloaded v4.1.5 fresh on this box: http 200, runs, tree-shakes to ~73KB minified for a kitchen-sink page, sub-1.5s) **+ base64 WOFF2 `@font-face` in the design tokens** so real fonts ship inline and the gate stays intact. Every stack still terminates in the existing `site_renders` chromium check — **zero new trust surface**.
- **CMS** = a `pages`/`blocks` Postgres content model inside Relay's own DB (the documented source of truth), NOT Directus/Payload/Strapi. Owner edit = one `UPDATE`; republish = re-enqueue the page's build task through the same `processTask → writeFileSync → verify(site_renders)` path. Defer to phase 3.

**One gate caveat I verified and closed:** Tailwind's minified output contains exactly one `https://` — the license comment `/* ...https://tailwindcss.com */`. The gate regex at `verify.ts:84` (`url(https?:` / `src=https?:` / `via.placeholder`) does NOT match a bare URL in a comment, so it passes today — but the inline step must strip the leading license comment so this stays robust, not accidental.

**Vendor binary is NOT yet present** (`tools/tailwindcss` is absent) — that download is roadmap step 1.

## Stacks

### Designed-Inline-HTML + Tailwind excellence layer (DEFAULT)
- **For:** marketing / landing, multi-page brochure (restaurant, clinic, salon, agency, studio, services), portfolio / case-studies, store / static catalog. This is the default for any brief the router cannot confidently classify as content/docs. Payments are explicitly out of scope in v1.
- **Agents emit:** Unchanged interface: the `build` agent emits ONE complete self-contained HTML document per page (full <!doctype html>, shared nav linking all pages). NEW: its role prompt (agents.ts:26) is rewritten to a tight design-system contract — use Tailwind utility classes on elements (no hand-rolled <style>), a fixed type scale (text-5xl md:text-7xl tracking-tight), spacing rhythm (py-24/32, max-w-5xl mx-auto), glass nav (backdrop-blur bg-*/70 border-b border-white/10), ring cards (ring-1 ring-white/10), gradient bg-clip-text headlines, glow shadows, and hover/transition motion. A new `design` department emits a strict JSON token contract {palette, type, scale, radii, shadows, motion, fontFaces} where fontFaces are base64 WOFF2 @font-face data-URIs so REAL fonts ship inline (fixes the YogaLX never-loads-Playfair bug) without an external <link>.
- **Build:** Deterministic post-build step in runner.ts, immediately before verify, exactly where the placeholder-swap already lives (lines 63-71): (1) write the agent's HTML to <slug>.html; (2) write app.css = `@import "tailwindcss";`; (3) run the vendored `tools/tailwindcss -i app.css -o out.css --content <slug>.html --minify` (~0.1-0.7s, zero network); (4) STRIP Tailwind's leading `/* ... https://tailwindcss.com */` license comment from out.css; (5) inline out.css into the page by replacing any <link>/app.css ref with a single <style>...</style> block; (6) delete app.css/out.css temp files. Result: one self-contained gate-safe file — byte-shaped exactly like today's builds. Multi-page already works; compile per page independently (shared nav unaffected). Vendor the binary ONCE to /root/agency-pipeline/tools/tailwindcss (chmod +x, commit) — verified: download is http 200, runs on this box, ~120MB one-time.
- **Verify:** REUSED UNCHANGED: `site_renders` (verify.ts:74-97) — size floor, valid <html>/<body> structure, the existing external-asset/placeholder regex (verify.ts:84), and the chromium screenshot >3000b. PLUS the existing `wcag` token check on the design dept's palette. TIGHTEN the gate (verify.ts:84) to ALSO reject any leftover `<link rel="stylesheet" href="http` or any remaining `app.css` reference, so a build that skipped the inline step fails deterministically instead of 404-ing in production. No new verify primitive needed.

### Eleventy (content SSG)
- **For:** blog / content / magazine / newsletter / editorial, and docs / documentation / API reference / SDK guide / handbook. These are exactly the archetypes where one-shot LLM HTML is weakest and where Markdown + an owned layout wins. NOT for marketing/brochure (that is Stack A).
- **Agents emit:** The `content` department emits Markdown + frontmatter (per page), and the `branding`/`design` department emits the same JSON token contract (incl. base64 WOFF2 fontFaces). The build agent no longer hand-writes HTML — strictly easier for the LLM and it removes the 1998 problem because layout/CSS quality is owned by the curated template, not the model.
- **Build:** NEW runner dispatch for Eleventy-routed projects: create /sites/<id>/_src/, SYMLINK a shared pre-installed node_modules (per-project footprint ~36K, no npm install — install is the only slow part and is pure waste since the toolchain never changes), drop the cached template's _includes/ (layout + tokens.css), write the agent's index.md/about.md/... into _src/, inject the branding JSON into tokens.css as CSS custom properties (and inline the WOFF2 @font-face), then run `./node_modules/.bin/eleventy --input=_src --output=/sites/<id>` (~0.6s, zero network). Flat `permalink: /<slug>.html` lands files EXACTLY where Relay already serves them — no serving change. Run the same Tailwind compile+inline on the produced pages so the excellence layer applies here too. Build the cached-template repo ONCE.
- **Verify:** NEW rule `ssg_builds`: run the Eleventy build via execFileSync; build MUST exit 0 (compile errors = hard fail; the agent gets stderr as `feedback` on retry — the loop already does retry-with-feedback at runner.ts:80-82). THEN reuse `site_renders` UNCHANGED on each produced <slug>.html (Eleventy output is inlined-<style>, 0 external https — verified structurally identical to today's gate-passing builds). `ssg_builds` is technically optional (site_renders already catches a failed build via missing/blank output) but worth adding for a precise stderr feedback signal on retry.

## Routing
Deterministic keyword router (NO LLM in the hot path), lives in planner.ts before validate(). `classifyArchetype(brief) -> archetype` via a ~40-keyword weighted scorer; the winning archetype maps to a stack via a STACKS registry {archetype -> {departments[], stack, buildCmd, inline:bool}}. Result is written to projects.params.archetype and params.stack and logged in run_events as the `planned` detail — this IS the owner's 'FORCED, assigned per build need': deterministic, reproducible, auditable.

Routing table (archetype -> stack | decision-signal keywords):
- marketing / landing -> Stack A (+Tailwind layer) | landing, launch, waitlist, saas, startup, app, beta, sign up
- multi-page brochure (DEFAULT) -> Stack A | restaurant, clinic, salon, agency, studio, services, menu, about us, booking
- blog / content -> Stack B Eleventy | blog, articles, posts, magazine, newsletter, editorial
- docs -> Stack B Eleventy (docs layout) | docs, documentation, api reference, sdk, guide, handbook
- portfolio -> Stack A | portfolio, case studies, showcase, photographer, freelance
- store / e-com (static catalog) -> Stack A | shop, store, ecommerce, cart, checkout, catalog (FLAG: payments out of scope v1)
- app-shell -> DEFERRED, route to brochure default | dashboard, login, account, realtime (revisit with Astro only if a real interactivity brief appears)

Fallback: when the keyword score is 0 (ambiguous brief), fall through to the existing LLM planner as a CLASSIFIER ONLY to pick the archetype; if that also fails, default to Stack A brochure. The LLM is never trusted to verify — only to classify when keywords are silent.

## Excellence layer
Concrete, gate-safe, deterministic — two parts, both proven on this box:

1. VENDORED TAILWIND STANDALONE CLI as a post-build compile-and-inline step (the single highest-ROI move). Download tailwindcss-linux-x64 ONCE to /root/agency-pipeline/tools/tailwindcss (verified: http 200, ~120MB, runs, tree-shakes a kitchen-sink page to ~73KB minified in <1.5s; a normal page is ~12-20KB / ~80-100ms). No node deps, no package.json, no CDN. The build agent keeps emitting one text artifact; a local binary turns it into a 2024 studio page. The agent CAN'T drift back to 1998 because the prompt prescribes the modern primitives (glass nav, ring cards, gradient bg-clip-text, glow shadows, type scale, motion) and Tailwind enforces consistent values.

2. REAL FONTS INLINE via base64 WOFF2 @font-face in the `design` department's JSON token contract. This directly fixes the YogaLX bug I verified — fonts referenced but never loaded — without ever emitting an external <link> that would trip the gate.

GATE-SAFETY (verified against the literal verify.ts:84 regex): Tailwind's minified output's ONLY https reference is the license comment `/* tailwindcss v4.x | MIT | https://tailwindcss.com */`. The gate regex matches `url(https?:` / `src=https?:` / `via.placeholder` — a bare URL inside a CSS comment does NOT match, so it passes TODAY. To make that robust rather than accidental, the inline step strips Tailwind's leading license comment before injecting the <style> block. Net cost: ~0.1s/page, zero new network deps at build time, fits the agent-text -> deterministic-verify model exactly.

ZERO-BINARY FALLBACK (pin, not lead): if the CLI ever can't run, inline a committed classless CSS file (Open Props ~30KB) whole. Keep this as the documented degradation path so the pipeline never hard-blocks on the binary.

## CMS
Build the editable layer as a Supabase/Postgres content model INSIDE Relay's existing DB — explicitly NOT a standalone headless CMS. A real headless CMS (Directus: verified ~70s install, ~779MB disk, ~300-500MB idle RSS) is a SECOND always-on Node server with its own auth/migrations/upgrades that solves the wrong half: it gives an admin UI + API but you STILL render the HTML yourself, so it does not remove Relay's render step — it just adds a server in front of it, and it fights the stateless-text-agent model (needs per-project admin tokens). Skip it.

Plan (defer to phase 3; do A+B first):
1. Schema (one migration in db/schema.sql, verified by the existing `sql_applies` rule): pages(id, project_id, slug, title, unique(project_id,slug)) and blocks(id, page_id, ord, type, data jsonb, updated_at).
2. Planner contract (planner.ts): the content/IA step emits an ordered block list {blocks:[{type:\"hero\",data:{...}}, ...]} (verify `json`). Add one `seed_blocks` task per page whose verify rule writes rows and asserts each `data` is valid jsonb with the type's required keys — a pure function over the JSON, no agent trust.
3. Render (agents.ts/runner.ts): the build agent renders the given blocks to one self-contained HTML page using the tokens; better, make it a DETERMINISTIC template per block type seeded by the agent's JSON, so the page is reproducible and the agent can't drift. Output still hits `site_renders` unchanged.
4. Editor + republish (small addition to server.ts): GET /api/site/:id/blocks -> ordered blocks; a tiny web/ editor edits block.data; POST /api/site/:id/blocks writes the UPDATE then RE-ENQUEUES that page's build task via the existing runLoop. Republish flows through the exact processTask -> writeFileSync -> verify(site_renders) path — an edited page is 'done' only when it re-renders non-blank. No new trust surface, zero new processes, ~0 extra RAM, two tables. The block-type templates ARE where the 'layers of excellence' design upgrade lands deterministically, so the design lens and the edit lens share one render contract.

## Roadmap
1. **Vendor the Tailwind binary + ship the design-system prompt contract + the deterministic compile-and-inline runner step (Stack A excellence layer). Download tailwindcss-linux-x64 to tools/tailwindcss (chmod +x); rewrite agents.ts:26 `build` role to the utility-class design contract; add the ~15-line compile->strip-license-comment->inline->cleanup step in runner.ts:63-71; add the base64-WOFF2 fontFaces token to a new `design` dept; tighten the gate (verify.ts:84) to also reject leftover <link href=http / app.css.** — Single highest-ROI move and the owner's #1 complaint ('1998 look'). It upgrades EVERY existing and future Stack-A build (the bulk of briefs) with one prompt + one ~15-line runner step + one gate assertion. No new stack, no new server, no per-project install, zero new network deps at build time. I verified end-to-end on this box: binary downloads (200), runs, tree-shakes, and the output is gate-safe (only https is the license comment, which the regex ignores and the strip step removes). Fixes the verified YogaLX never-loads-fonts bug. Ship this first, alone, and the product looks 2024 the same day.
2. **Add the deterministic keyword router + STACKS registry in planner.ts (writes params.archetype/params.stack, logged in run_events). Default everything to Stack A for now — no Stack B behavior change yet.** — Establishes the owner's 'FORCED, assigned per build need' routing as a deterministic, auditable decision BEFORE adding a second stack to route to. Cheap (one classifier function + a registry object), no agent trust, and it makes the next step a pure dispatch addition rather than a rewrite. Low risk because every route still lands on the already-shipped Stack A.
3. **Build the cached Eleventy template repo (shared node_modules, _includes layout mapping tokens->CSS vars + inline WOFF2) and wire Stack B: runner dispatch (symlink node_modules, write .md, run eleventy) + the `ssg_builds` verify rule. Route blog/content/docs archetypes to it.** — Eleventy only earns its keep on blog/docs, where one-shot LLM HTML is weakest and Markdown is what MiniMax does best. It's a meaningful but contained add (one cached template + symlink scaffold + one optional verify rule) and it reuses site_renders unchanged. Doing it AFTER the router means it's a registry entry + a dispatch branch, not surgery. Deferring it past step 1-2 is correct: the excellence layer already lifts these sites' quality, so the SSG is an enhancement, not a blocker.
4. **Add the pages/blocks Postgres content model + editor endpoints + republish-via-re-enqueue (the CMS/editable layer).** — Owner-editability is real value but it is NOT what makes builds look 2024, and it touches the most surfaces (schema, planner contract, render, server). Sequencing it last lets steps 1-3 stabilize the render contract that the block templates depend on, and it reuses sql_applies/json/site_renders with zero new verify primitives and zero new processes. Pure additive, restart-safe by construction.
5. **DEFER: Astro and any real headless CMS (Directus/Payload/Strapi/Keystone). Route app-shell briefs to the Stack A brochure default for now.** — Astro buys nothing over Stack A/B for static brochure/content sites (4x slower install, 7x larger, needs a community single-file integration) and only pays off for genuine islands/interactivity that no current brief requires. A headless CMS is a second always-on server (~300-500MB idle, own auth/upgrades) that doesn't even remove the render step. Adding either now is cost without payoff and risks the build-once mandate. Revisit ONLY when a concrete interactivity brief (search-as-you-type, embedded React) actually arrives.
